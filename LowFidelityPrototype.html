Design Concepts

During the brainstorming we listed three major components of a bill splitting service. The Login/Registration section, the Individual Bill View/Creation section, and the List of all Bill section. We decided to treat these three sections as three individual Low-Fidelity prototypes with the following Usability goals.

Login/Registration

  1. Remember Me
  2. Password Reset
  3. E-mail as Username
  4. Basic contact information
  5. Social media registration
  6. Optional profile picture
  7. Ability to register without payment method
  8. Option to register multiple payment methods
  9. Ability to edit profile information
  10. Ability to delete account
  11. Link to access profile from any page in website

Individual Bill view/Bill creation

  1. Bill Information
    • Cost
    • Date of Creation
    • Member Contribution
    • Preferred Payment
    • Text Description
    • Category/group
  2. Add proof of bill
  3. Add payers
  4. Submit bills
  5. Catagorize expenses into subgroups
  6. Add descriptions to expenses/subgroups
  7. Specify the contribution of each payer
  8. Indicate if someone has paid
  9. Comment section
  10. Make the bill easily shareable and with a perma-link

List of Bills

  1. Create, modify, delete, hide a Bill
  2. Sort bills by catagory
  3. Search function
  4. Option to pay quickly from the website

Separating the Bill splitting service into three parts allowed us to focus on the main actions the user will need to do. Using the 10 + 10 method we came up with the sketches you see below:

From these sketches we have found three themes. The Bubble design which organizes information in bubbles that can be moved around by gesture. The Card design which organizes bills as a list of tabs as one would sort a file drawer. And finally the Spreadsheet design which mimicks the popular spreadsheet template to organize the bills and expenses in those bills in tables. Then, made 3 pencil and paper prototypes that would embody these themes.

Usability Goals

Navigation

Navigating through the different menus of the system should not require explanation. We are aiming to simplify the transaction management process for students. The design should enable the user to navigate through menus and locate functions unaided. If the user is constantly asking for help or is unable to perform a simple action, then the design is hindering the functionality.

Login / Register

The user should be able to gain access to the application without assistance. Most user will be exposed to system through recommendations, or being added as a guest payee. It is important that a user’s first exposure to the system doesn’t dissuade them from registering. Registration itself should be simple and capture the relevant information.

Bill Listing

The user should be able to find and organize their bills. The bill listing should make it easy for the user to recognize what needs to be paid and who needs to contribute. It is important that the user has a clear overview of their finances to have all the information in an easily understandable manner. Going through the list should be seamless and finding a target bill quick.

Bill Creation

The system should streamline the process of creating and tracking bills. Creating a bill should be easier than the existing alternatives. The process should allow the user to detail their expenditure, split the costs among friends and continually update the bill as payments are made.

Bill View

The system should provide a consistent source of information for both payer and payee. Making the individual bills clear and concise is paramount to the system, this is what they are going to be replacing their current bill format with. Users should be able to know exactly what they need to pay and why. Paying itself through the system should be a one-click task.

    Summary of Test Results

    Feedback for Bubble Prototype

    The Bubble prototype’s conceptual model was hard to understand. Navigation was difficult. Color coding really help to identify the status of bill. Easier for them to locate the rent because it was clearly marked. Sometimes external user was lumped up with roommates. Field popped up after clicked was quick and very helpful. The second time the user went through the workflow was a lot quicker and the user was notably more confident. This shows that the bubble prototype helps users recognize, diagnose, and recover from errors. Design was the most aesthetically pleasing and minimalistic design.

    Feedback for Card Prototype

    User was quick to find the register section upon first use. Would have liked option to connect using social media. Users found it easy to navigate through the list of cards. Users were unclear what the negative sign represented. Not sure how to get from the card list to the individual bill. People responded well to the overall design. Users would prefer the cards to give more information at first glance. Second visual element needed to distinguish bills. Cards and text is clear but it needs to be supplement with graphic cues. Add small bubbles with payer faces (like facebook message). Color coding would help a lot. Users were confused with the guest user input field, they were not sure if it was mandatory or not. Making an individual bill required a lot of input from the user, bill creation process should be streamlined. Users were able to do all the tasks at a decent speed on the first try and they were very quick on the second time attempting the workflow.

    Feedback for Spreadsheet Prototype

    Clear Login button. Too much information on screen. Recognizable design but students don’t need to detail their expenses as much as professionals. Could not parse information at a glance, must go through all the steps. Terminology of the header confusing. When creating a bill, users did not know where to start - No clear place to put inputs and where they should go. Users were not sure where to click. Thought they could modify values from bill list. Did not need so much information about individual expenses. Did not understand that only users registered in the system would get automatically notified and external users needed an extra step. Work-flow after was quick.

    Design going forward

    User was much quicker on the second pass. We should have a quick start guide to alleviate first-use speed bumps. Proof of payment may be superfluous. Focus on speed of making a new bill maybe by adding option to input more details later if needed. Maybe long-term (rent) vs short-term (night out) bill creation. We should use color. Users prefer to sort things by the names they give them and not macro-information like date of creation or value of the bill. Need to make distinct known contact vs guest user. Guest user information pop-up could alleviate drawback in card sort where user would not like to leave the guest user field blank. We should clear confusion between bills you created and bills you must pay. Without any guidance, users quickly feel overwhelmed. Makes sense to sacrifice flexibility for smoother user interface. Bill list and individual bill view should have separate layouts in order to prevent confusion between both. Going further we should keep doing iterative design on the card prototype. We will make the cards clearer by using elements from the bubble design users responded to. We can use the spreadsheet design as an example of design pitfalls to avoid. Notably: Putting too much information on screen, giving the users too many options and not guiding the user through the payment process. We have found that for our user base, we must concentrate on smooth webpage navigation and clarity of information on screen.
